**<purpose> section:**
Execute a design phase of Architecture GSD. Given a phase number, coordinate the full pipeline: run arch-researcher (if no RESEARCH.md exists), run arch-roadmapper (if no ROADMAP.md exists), enter the arch-planner + arch-checker bounded revision loop (max 3 iterations), then spawn arch-executor agents in parallel per wave assignment. Updates STATE.md after every plan completion. This workflow is the single entry point for progressing through design phases — the human runs `/arch-gsd:execute-phase N` and the pipeline produces design documents autonomously.

**<process> section with 10 steps:**

**Step 1: Initialize and Orient**
Extract phase number from $ARGUMENTS. Read .arch/STATE.md to understand current position and accumulated decisions. Read .arch/CONTEXT.md to load system description (domain, actors, scale, constraints, locked-decisions). Validate both exist — if .arch/CONTEXT.md does not exist, display: "No CONTEXT.md found. Run /arch-gsd:new-system first to initialize the project." and stop.

**Step 2: Check Prerequisites**
Check if .arch/RESEARCH.md exists:
  - If not: spawn arch-researcher via Task(model="sonnet", prompt="Read agents/arch-researcher.md for instructions. System context: .arch/CONTEXT.md. Produce .arch/RESEARCH.md.")
  - Wait for completion. If status is "failed", display error and stop.

Check if .arch/ROADMAP.md exists:
  - If not: spawn arch-roadmapper via Task(model="opus", prompt="Read agents/arch-roadmapper.md for instructions. System context: .arch/CONTEXT.md, .arch/RESEARCH.md. Produce .arch/ROADMAP.md.")
  - Wait for completion. If status is "failed", display error and stop.

If both exist, skip to Step 3.

**Step 3: Locate Phase in Roadmap**
Read .arch/ROADMAP.md. Find the phase entry matching the requested phase number. Extract: phase name, goal, success criteria, requirements list, artifact list. If phase not found: display "Phase {N} not found in ROADMAP.md. Available phases: {list}" and stop.

Display to human: "Executing Phase {N}: {phase_name}\nGoal: {goal}\nArtifacts: {count} design documents"

**Step 4: Create Phase Directory**
Create the phase working directory if it does not exist:
  Bash: mkdir -p .arch/phases/{phase-slug}

**Step 5: Run arch-planner**
Spawn arch-planner via Task(model="sonnet", prompt="Read agents/arch-planner.md for instructions. Phase: {N} — {phase_name}. ROADMAP: .arch/ROADMAP.md. CONTEXT: .arch/CONTEXT.md. RESEARCH: .arch/RESEARCH.md. Phase directory: .arch/phases/{phase-slug}/. Produce PLAN.md files.")

Wait for completion. Parse return. If status is "failed", display error and stop.

**Step 6: Run arch-checker (Bounded Revision Loop)**
```
iteration = 0
max_iterations = 3

while iteration < max_iterations:
  Spawn arch-checker via Task(model="haiku", prompt="Read agents/arch-checker.md for instructions.
    Phase directory: .arch/phases/{phase-slug}/. ROADMAP: .arch/ROADMAP.md. CONTEXT: .arch/CONTEXT.md.
    Check all PLAN.md files in the phase directory.")

  Wait for completion. Parse return.

  if status == "passed":
    Display: "Plans approved by arch-checker (iteration {iteration+1})"
    break

  if status == "issues_found":
    iteration += 1
    if iteration >= max_iterations:
      # ESCALATION — do NOT silently proceed
      Display structured gap report to human:
        "arch-checker found unresolved issues after {max_iterations} revision iterations.
        STOPPING execution — human review required.

        Unresolved issues:
        {format each issue: plan, dimension, severity, description, fix_hint}

        To continue:
        1. Review and manually fix the flagged PLAN.md files
        2. Re-run /arch-gsd:execute-phase {N}"
      STOP. Do not proceed to Step 7.

    # Re-invoke arch-planner in REVISION MODE
    Spawn arch-planner via Task(model="sonnet", prompt="Read agents/arch-planner.md for instructions.
      REVISION MODE — targeted updates only.
      Issues from arch-checker: {format issues as YAML}
      Phase directory: .arch/phases/{phase-slug}/.
      Fix the flagged issues. Do not rewrite entire plans.")
    Wait for completion.
```

**Step 7: Execute Plans by Wave**
Read all PLAN.md files from the phase directory. Extract wave assignments from frontmatter. Group plans by wave number.

For each wave (in ascending order):
  Display: "Wave {W}: Executing {count} plans in parallel..."

  For each plan in this wave:
    Spawn arch-executor via Task(model="sonnet", prompt="Read agents/arch-executor.md for instructions.
      Execute plan: .arch/phases/{phase-slug}/{plan-filename}.
      CONTEXT: .arch/CONTEXT.md. RESEARCH: .arch/RESEARCH.md.
      References: @references/agent-spec-format.md @templates/agent-spec.md @templates/event-schema.yaml @templates/failure-modes.md
      Validate after writing: node bin/arch-tools.js detect-stubs {output_file} && node bin/arch-tools.js validate-names {output_file}")

  Wait for ALL arch-executor agents in this wave to complete.

  For each completed agent:
    Parse return. If status is "human_needed" (deviation Rule 4): display the error and STOP.
    If status is "gaps_found": collect gaps for later reporting.
    If status is "failed": display error, mark plan as failed, continue with remaining plans.

  Spot-check: For 1-2 completed documents, run `node bin/arch-tools.js detect-stubs {file}` to verify.

  Display: "Wave {W} complete. {success_count}/{total_count} plans succeeded."

Proceed to next wave.

**Step 8: Handle Deviation Flags**
After all waves complete, review auto_flagged items from arch-executor returns. If any exist:
  Display: "arch-executor auto-flagged {count} items during execution:

Read current .arch/STATE.md. Update:

  - Last activity: {date} — "Phase {N} executed: {count} plans, {success_count} complete"

  - Trim STATE.md to 100 lines maximum — remove oldest decision entries if approaching limit

Write updated .arch/STATE.md using the Write tool. Commit: `git add .arch/STATE.md && git commit -m "docs: update STATE.md after phase {N} execution"`


**Specialized agent role clarification (context-engineer, schema-designer, failure-analyst):**
These three agents are NOT directly spawned by execute-phase. They are **reference specs consumed by arch-executor**. When arch-planner assigns a task to produce events.yaml, arch-executor reads `agents/schema-designer.md` as domain guidance for how to produce that document (typing rules, derivation logic, naming conventions). When assigned a CONTEXT-FLOWS.md task, arch-executor reads `agents/context-engineer.md` for bottleneck analysis methodology. When assigned a FAILURE-MODES.md task, arch-executor reads `agents/failure-analyst.md` for banned-phrase enforcement rules and failure cataloging structure. The execute-phase workflow passes these spec paths to arch-executor in the Task() prompt (Step 7) as part of the references list. This design keeps the orchestrator lean while giving arch-executor domain-specific production guidance.
