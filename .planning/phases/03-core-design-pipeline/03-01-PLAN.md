**<role> section:**
Spawned by /arch-gsd:execute-phase after arch-researcher completes (Wave 1 output consumed). This agent derives the design phase structure from .arch/CONTEXT.md and .arch/RESEARCH.md. It produces .arch/ROADMAP.md — a phased design roadmap with phase names, goals, success criteria, dependency ordering, and per-phase artifact lists. The roadmap is the contract that arch-planner decomposes into tasks and arch-executor produces documents against. arch-roadmapper adapts the number of design phases based on complexity signals: systems with scale.agents < 5 use a 3-phase structure; scale.agents 5-15 use 5-phase; scale.agents > 15 use 7-phase with additional integration phases (per research recommendation).

**<upstream_input> section:**
- Reads this spec from agents/arch-roadmapper.md — loaded by execute-phase orchestrator
- Reads .arch/CONTEXT.md — uses domain (determines architectural style), actors (agent candidates), scale (complexity signal for phase count), constraints (technology boundaries), locked-decisions (non-negotiable design choices). The domain field is the primary structural signal.
- Reads .arch/RESEARCH.md — uses "Architecture Patterns" section for design phase templates, "Standard Stack" for technology placement, "Open Questions" to identify phases that need human checkpoints
- Reads .arch/STATE.md — uses Current Position to understand design progress and Decisions for constraints

**<downstream_consumer> section:**
- arch-planner reads .arch/ROADMAP.md — uses phase entries (goal, success criteria, requirements list, artifact list) to decompose each phase into executable tasks with wave assignments
- arch-checker reads .arch/ROADMAP.md — uses success criteria and requirement mappings to validate that arch-planner's tasks cover all requirements for each phase
- execute-phase workflow reads .arch/ROADMAP.md — uses phase ordering and dependency information to determine execution sequence
- arch-executor reads .arch/ROADMAP.md — uses phase artifact lists to confirm which design documents to produce

**<execution_flow> section — 10 steps:**

Step 1: Read .arch/STATE.md to orient — extract current position, locked decisions, and any blockers.

Step 2: Read .arch/CONTEXT.md to load the full system description. Extract: domain, actors (count and names), scale.agents (complexity signal), constraints, locked-decisions, non-goals.

Step 3: Read .arch/RESEARCH.md to load architecture patterns and technology recommendations. Extract: recommended patterns, standard stack, open questions.

Step 4: Determine phase count from complexity signals:
  - If scale.agents is an integer: use the integer directly
  - If scale.agents is a range (e.g., "10-20"): use the upper bound
  - Agent count < 5: 3-phase structure (Schema + Agents, Integration + Topology, Verification)
  - Agent count 5-15: 5-phase structure (Context + Research, Schema Design, Agent Contracts, Topology + Context Flows, Failure Modes + Integration)
  - Agent count > 15: 7-phase structure (adds separate phases for subsystem grouping and cross-subsystem integration)

Step 5: For each design phase, derive:
  - Phase name: descriptive, kebab-case (e.g., "schema-design", "agent-contracts", "topology-and-flows")
  - Goal: outcome statement from the USER perspective (not task-shaped)
  - Success criteria: 3-5 testable truths that must hold when the phase is complete
  - Requirements: which PIPE-xx, SPEC-xx, OUTP-xx requirements this phase addresses
  - Artifacts: which design documents (specific paths) this phase produces
  - Dependencies: which earlier phases must complete first

Step 6: Validate the phase structure:
  - Every requirement in CONTEXT.md's domain must map to at least one phase
  - Every output document type (agents/, events/, topology/, context/, failure/) must appear in at least one phase
  - Dependencies form a DAG (no circular phase dependencies)
  - Locked decisions are honored (if CONTEXT.md locks "event sourcing", event schema design phase must include event sourcing patterns)

Step 7: Compute the wave structure within each phase using the artifact dependency rules from research:
  - Wave 1: research + roadmap outputs (no artifact dependencies)
  - Wave 2: schemas + context flows (depend on research)
  - Wave 3: agent contracts (depend on schemas for event names)
  - Wave 4: topology + failure modes (depend on agent contracts)
  Note: Wave assignments are per-phase recommendations. arch-planner refines them into task-level waves.

Step 8: Write .arch/ROADMAP.md using the Write tool. Structure:
  - Header: "# Design Roadmap: {system name from CONTEXT.md domain}"
  - Metadata: system domain, phase count, total design documents
  - Phase entries: each phase with all fields from Step 5
  - Dependency graph: text-based DAG showing phase ordering
  - Artifact index: every design document path with its producing phase

Step 9: Validate by checking: `node bin/arch-tools.js detect-stubs .arch/ROADMAP.md` returns clean: true. Verify all actors from CONTEXT.md appear in at least one phase's scope.

Step 10: Return structured JSON result to orchestrator.
