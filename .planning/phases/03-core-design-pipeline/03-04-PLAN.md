**<role> section:**
Spawned by /arch-gsd:execute-phase for each task in an approved PLAN.md after arch-checker passes the plan. Step 1: Read .arch/STATE.md to orient — extract current position, decisions, and any active concerns.

  - `design/agents/*.md` → agent contract (use templates/agent-spec.md + references/agent-spec-format.md)
  - `design/events/events.yaml` → event schema registry (use templates/event-schema.yaml)
  - `design/topology/*.md` → topology diagram
  - `design/context/*.md` → context flow map
  - `design/failure/*.md` → failure mode catalog (use templates/failure-modes.md)

Step 4: Read the appropriate template and reference documents based on document type. For agent contracts, read templates/agent-spec.md (section scaffold + HTML comment verification rules) and references/agent-spec-format.md (required content depth per section + COMPLETE/INCOMPLETE examples).

Step 5: Read .arch/CONTEXT.md for system context — domain, actors, constraints, locked-decisions. Read .arch/RESEARCH.md for architecture patterns and implementation guidance relevant to this document. If document type is agent-contract and design/events/events.yaml exists, read it for event names to cross-reference.

Step 6: Write the document to the path specified in <files>. Follow the task's <action> instructions as primary guidance. Apply the template structure. Produce dual-format content:

  For agent contracts:
  - 7 required sections: Role, Upstream Input, Downstream Consumer, Execution Flow, Structured Returns, Failure Modes, Constraints
  - Each section has markdown prose PLUS embedded YAML canonical block
  - The YAML block contains machine-checkable fields (agent name, model, inputs list, outputs list, event references)
  - Use the HTML comment verification rules from templates/agent-spec.md as minimum content expectations
  - Preserve existing stub frontmatter (name, model, tools, color) — only replace body content

  For event schemas:
  - Pure YAML format (already canonical — no dual format needed)
  - Every event: name (PascalCase), type (event|command), version, description, payload (typed fields), producers, consumers, error_cases
  - Every payload field: name, type (from allowed types only), constraints, example, required
  - Optional fields must have: default
  - Banned types: any, object (without fields), data, mixed, unknown, arbitrary

  For failure mode catalogs:
  - 3 required sections: Failure Mode Catalog, Integration Point Failures, Residual Risks
  - Each failure mode: Trigger, Manifestation, Severity, Recovery (Immediate + Escalation), Detection
  - Banned phrases in Recovery: "handles gracefully", "escalates as needed", "retries appropriately"

  For topology:
  - Mermaid diagram in markdown code block for agent dependency graph
  - Table of communication channels (from-agent, to-agent, mechanism, data-type)
  - Dual-format YAML block with adjacency list

  For context flows:
  - Per-agent context table (reads, writes, passes downstream)
  - Information bottleneck analysis (which agents are context-starved?)
  - Dual-format YAML block with context injection map

Step 7: Apply deviation rules (from deviation_rules section):
  - Check Rule 3: Does CONTEXT.md have requirements relevant to this task not mentioned in <action>? If yes, address them and flag.
  - Check Rule 4: Does CONTEXT.md contradict anything in the task? If yes, STOP and return human_needed.

  - For agent contracts: verify all 7 sections are present (grep for section headers)
  - For event schemas: verify all events have required fields (name, type, payload, producers, consumers, error_cases)

**<failure_modes> section — 4 modes:**

FAILURE-01: PLAN.md Task Not Found
  - Trigger: The task name or number specified by the orchestrator does not match any task in the PLAN.md file
  - Manifestation: arch-executor cannot determine what to write — no <action>, <files>, or <verify> available
  - Severity: critical
  - Recovery: Immediate — return { status: "failed", error: "Task '{task_id}' not found in {plan_path}" }. Escalation — orchestrator surfaces error to human; likely a plan numbering mismatch
  - Detection: Task extraction at Step 2 finds no matching <task> block

FAILURE-02: Template or Reference Document Missing
  - Trigger: templates/agent-spec.md, templates/event-schema.yaml, or references/agent-spec-format.md does not exist at expected path
  - Manifestation: arch-executor cannot determine required sections, content expectations, or typing rules — would produce documents that fail verification
  - Severity: high
  - Recovery: Immediate — if only reference docs are missing, write document using <action> instructions only (degraded but functional). If template is missing, return { status: "failed", error: "Template not found: {template_path}" }. Escalation — human re-runs Phase 1 to restore missing templates
  - Detection: fs.existsSync check at Step 4 returns false

FAILURE-03: detect-stubs Finds Stubs After Writing
  - Trigger: After writing the document, `node bin/arch-tools.js detect-stubs` reports stubs_found > 0
  - Manifestation: Document contains banned phrases (TBD, handles gracefully, as needed) — would fail Level 2 verification
  - Severity: medium
  - Recovery: Immediate — for each stub finding, rewrite the flagged section with concrete content. Re-run detect-stubs. Allow up to 2 correction iterations. Escalation — if stubs persist after 2 corrections, return { status: "gaps_found", output: "{file}", gaps: ["Stub phrases remain in sections: {list}"] }
  - Detection: detect-stubs returns clean: false at Step 8

FAILURE-04: Context Window Exceeded During Document Writing
  - Trigger: Writing a complex agent contract (100+ lines of dual-format content) plus reading templates and references exhausts the context window
  - Manifestation: Later sections of the document are shallow, vague, or truncated compared to earlier sections
  - Severity: high
  - Recovery: Immediate — write all sections even if later ones are shorter. Run detect-stubs to identify any degradation. If degraded sections are detected, return { status: "gaps_found", output: "{file}", gaps: ["Sections {X, Y} may be degraded due to context pressure"] }. Escalation — orchestrator can split the task or re-invoke arch-executor with narrower scope
  - Detection: Observable as output quality degradation — detect-stubs may catch some issues, but context exhaustion produces "technically compliant but shallow" content that detect-stubs misses. Manual review recommended for documents over 150 lines.
