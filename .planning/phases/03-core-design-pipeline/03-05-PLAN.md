**<role> section:**
Spawned by /arch-gsd:execute-phase as a specialized Wave 2 agent (after arch-researcher and arch-roadmapper complete). This agent designs the canonical event and command schema registry for the target system. It produces design/events/events.yaml — the single source of truth for all event and command names, typed payloads, and producer/consumer annotations. events.yaml is the vocabulary that every downstream design document references: agent contracts use event names in their upstream/downstream sections, topology uses events to draw communication channels, and arch-verifier (Phase 4) resolves all event references against this file. schema-designer runs BEFORE agent contracts are written (Wave 2 before Wave 3) specifically because agent contracts must reference event names that already exist in events.yaml.

**<upstream_input> section:**
- Reads this spec from agents/schema-designer.md — loaded by execute-phase orchestrator
- Reads .arch/CONTEXT.md — uses actors (interaction sources that generate events), domain (architectural style determines event patterns), scale (throughput/latency affect payload sizing), constraints (technology limits on event size/format), locked-decisions (may mandate specific event patterns like event sourcing)
- Reads .arch/RESEARCH.md — uses "Architecture Patterns" for event pattern selection (pub/sub, event sourcing, command-query), "Standard Stack" for serialization format guidance
- Reads .arch/ROADMAP.md — uses agent list from the design roadmap to derive producer/consumer annotations (which agent produces each event, which consumes it)
- Reads templates/event-schema.yaml — uses field structure, allowed/banned types, naming rules, error_cases requirements as the production template
- Reads references/naming-conventions.md — uses PascalCase for events, SCREAMING_SNAKE for commands naming rules

**<downstream_consumer> section:**
- arch-executor reads design/events/events.yaml — uses event names and typed payloads when writing agent contract Upstream Input and Downstream Consumer YAML blocks
- arch-verifier (Phase 4) reads design/events/events.yaml — uses as canonical registry to resolve all event references across all design documents (Level 4 verification)
- failure-analyst reads design/events/events.yaml — uses error_cases section to inform failure mode analysis for each event consumer
- context-engineer reads design/events/events.yaml — uses producer/consumer annotations to map information flow between agents

**<execution_flow> section — 10 steps:**

Step 1: Read .arch/STATE.md to orient.

Step 2: Read .arch/CONTEXT.md. Extract actors (event sources), domain (event pattern guidance), scale, locked-decisions (may mandate event patterns).

Step 3: Read .arch/RESEARCH.md. Extract architecture patterns relevant to event design. Identify recommended event patterns (pub/sub, event sourcing, command-query separation).

Step 4: Read .arch/ROADMAP.md. Extract the full agent list across all design phases. Each agent is a potential event producer or consumer.

Step 5: Read templates/event-schema.yaml to load the production template with all field requirements and typing rules.

Step 6: Derive events from actor interactions. For each actor in CONTEXT.md:
  - What actions does this actor initiate? Each action becomes a command (SCREAMING_SNAKE).
  - What state changes result from these actions? Each state change becomes an event (PascalCase: action verb + "ed" or "Completed" / "Failed").
  - Example: Actor "Human Architect" initiates "new system design" → command INIT_DESIGN → events DesignInitiated, DesignCompleted, DesignFailed.

Step 7: For each derived event and command, specify:
  - name: PascalCase for events, SCREAMING_SNAKE for commands
  - type: "event" or "command"
  - version: "1.0"
  - description: One sentence, no TBD or vague language
  - payload: Array of typed fields. For each field:
    - name: snake_case field name
    - type: From allowed types ONLY (string, integer, float, boolean, array<T>, object{field: T}, enum[v1,v2])
    - constraints: Specific validation rules (max_length, min, max, pattern)
    - example: Concrete value demonstrating the field
    - required: true or false
    - default: Required if required=false
  - producers: Array of kebab-case agent names that emit this event/command
  - consumers: Array of kebab-case agent names that handle this event/command
  - error_cases: At least one per event/command with: condition (specific), payload_field, recovery (concrete)

Step 8: Validate naming conventions: every event name must pass PascalCase regex `^[A-Z][a-zA-Z0-9]*$`, every command must pass SCREAMING_SNAKE regex `^[A-Z][A-Z0-9_]*$`. No banned types (any, object without fields, data, mixed, unknown).

Step 9: Write design/events/events.yaml. Run validation: `node bin/arch-tools.js validate-names design/events/events.yaml` — fix any violations. Run: `node bin/arch-tools.js detect-stubs design/events/events.yaml` — fix any stub phrases.

Step 10: Return structured JSON result.

**context-engineer.md (name=context-engineer, model=opus, color=teal):**

<role>: Spawned by /arch-gsd:execute-phase as a specialized agent. Designs the context flow maps for the target system — what data each agent reads, writes, and passes downstream. Produces design/context/CONTEXT-FLOWS.md, a structured document exposing information bottlenecks and redundant handoffs. The context flow map is critical for identifying agents that are context-starved (need data not currently passed to them) and agents that receive redundant data (wasteful context injection).

<upstream_input>:
- .arch/CONTEXT.md — uses actors (interaction sources), constraints (boundaries), scale (data volume)
- .arch/ROADMAP.md — uses agent list and phase structure to enumerate all agents whose context needs must be mapped

<downstream_consumer>:
- arch-executor reads CONTEXT-FLOWS.md — uses per-agent context requirements when writing agent contract Upstream Input sections

- topology references CONTEXT-FLOWS.md — uses information flow data for communication diagram

<execution_flow> — 8 steps:
1. Read STATE.md, CONTEXT.md (actors, constraints, scale)
2. Read ROADMAP.md to enumerate all agents in the design
3. Read events.yaml (if exists) for event-based data flow

5. Build context injection map: for each agent, what is the minimum context it needs? What is its maximum useful context?
