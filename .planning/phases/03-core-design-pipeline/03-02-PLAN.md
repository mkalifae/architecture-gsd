**<execution_flow> section — 12 steps (the core wave assignment algorithm):**

Step 1: Read .arch/STATE.md to orient — extract current position, decisions, blockers.

Step 2: Read .arch/ROADMAP.md to load the current phase entry. Extract: phase name, goal (outcome statement), success criteria, requirements list, artifact list (specific design document paths), wave recommendations.

Step 3: Read .arch/CONTEXT.md to load locked-decisions and constraints that affect task specifications.

Step 4: Read .arch/RESEARCH.md to load "Don't Hand-Roll" items and "Common Pitfalls" for preventive task actions.

Step 5: Read references/agent-spec-format.md and templates/agent-spec.md to understand the required sections and content expectations for agent contract tasks. Read templates/event-schema.yaml for event schema task requirements. Read templates/failure-modes.md for failure mode task requirements.

Step 6: Decompose the phase artifact list into individual tasks. Each artifact path becomes one task. For each task, determine:
  - Document type: agent-contract, event-schema, topology, context-flows, failure-modes
  - Required sections: from the relevant template
  - Naming constraints: PascalCase events, kebab-case agents, SCREAMING_SNAKE commands
  - Cross-references: which other documents this task must reference

Step 7: Compute wave assignments using artifact dependency rules:
```
ARCHITECTURE_DEPENDENCY_RULES = {
  'event-schema': [],                        # No design artifact dependencies
  'context-flows': ['roadmapper-output'],    # Needs agent list from roadmapper
  'agent-contract': ['event-schema'],        # Needs event names to reference
  'failure-modes': ['agent-contract'],       # Needs agents to enumerate failure points
  'topology': ['agent-contract'],            # Needs all agent contracts to draw graph
}

for each task:
  deps = ARCHITECTURE_DEPENDENCY_RULES[task.document_type]
  if deps is empty:
    task.wave = 1
  else:
    task.wave = max(wave of tasks producing dep artifacts) + 1
```

Step 8: Group tasks into PLAN.md files following the rules:
  - Max 3 tasks per plan
  - Same-wave tasks with no file conflicts go in the same plan
  - Tasks producing the same document type can share a plan if ≤ 3
  - Cross-wave tasks go in separate plans with depends_on linking them

Step 9: For each plan, derive must_haves using goal-backward methodology:
  - truths: "What must be TRUE for this plan's output to be useful?" (3-5 observable behaviors from user perspective)
  - artifacts: each task's output file with min_lines, required content markers (contains: "<role>" for agent specs, contains: "payload:" for event schemas)
  - key_links: critical connections between this plan's output and other documents (e.g., agent contract -> events.yaml reference)

**<failure_modes> section — 4 modes:**

FAILURE-01: ROADMAP.md Phase Entry Missing or Incomplete
  - Trigger: Current phase number does not match any entry in .arch/ROADMAP.md, or the matching entry has no artifact list
  - Manifestation: arch-planner cannot decompose — no artifacts to assign to tasks
  - Severity: critical
  - Recovery: Immediate — return { status: "failed", error: "Phase {N} not found in ROADMAP.md or has no artifact list" }. Escalation — orchestrator re-runs arch-roadmapper to regenerate ROADMAP.md
  - Detection: ROADMAP.md parsing at Step 2 finds no matching phase entry

FAILURE-02: Wave Assignment Creates Circular Dependency
  - Trigger: Artifact dependency rules create a cycle (e.g., A depends on B which depends on A) — theoretically impossible with the fixed ARCHITECTURE_DEPENDENCY_RULES but could occur with locked-decision overrides
  - Manifestation: Wave assignment loop does not terminate or produces wave numbers that exceed the task count
  - Severity: high
  - Recovery: Immediate — detect cycle by checking if any task's wave exceeds total task count. Break the cycle by assigning the conflicting tasks to the same wave and documenting the forced parallel execution. Escalation — return { status: "gaps_found", gaps: ["Circular dependency detected between {A} and {B} — forced to same wave"] }
  - Detection: max(waves) > total_tasks after Step 7

FAILURE-03: Too Many Tasks for a Single Design Phase
  - Trigger: Phase artifact list contains more than 15 design documents (e.g., a system with 20+ agents means 20+ agent contract tasks)
  - Manifestation: Decomposition produces more than 5 PLAN.md files, which strains orchestrator context for tracking
  - Severity: medium
  - Recovery: Immediate — group related tasks aggressively (e.g., 2-3 agent contracts per plan if they don't share event dependencies). Cap at 5 plans per phase. Escalation — if grouping still exceeds 5 plans, suggest to orchestrator that the phase be split into sub-phases (e.g., "Agent Contracts A-J" and "Agent Contracts K-T")
  - Detection: Task count > 15 after Step 6

FAILURE-04: Locked Decision Conflicts with Artifact Dependency Rules
  - Trigger: A locked-decision in CONTEXT.md mandates a design choice that violates wave ordering (e.g., "topology must be designed before individual agents" — inverting the standard dependency)
  - Manifestation: arch-planner cannot satisfy both the locked decision and the standard wave ordering
  - Severity: high
  - Recovery: Immediate — honor the locked decision (non-negotiable). Override the standard wave ordering for the conflicting tasks. Document the override in the affected PLAN.md: "Wave order modified per locked decision: {decision}". Escalation — return status "complete" but include a warning: "Standard wave ordering overridden by locked decision — arch-checker should verify correctness"
  - Detection: Locked decision at Step 3 explicitly contradicts an entry in ARCHITECTURE_DEPENDENCY_RULES
