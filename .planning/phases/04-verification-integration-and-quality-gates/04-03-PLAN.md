<task type="auto">
  <name>Task 1: build-graph command and extractYamlBlocks helper</name>
  <files>bin/arch-tools.js</files>
  <action>
Add the build-graph command and supporting functions to arch-tools.js. This is the foundation for all Level 4 checks — build the graph once, query it multiple times.

1. Add `extractYamlBlocks(content)` helper function:
   - Uses regex `/```yaml\n([\s\S]+?)```/g` to extract all YAML code blocks from markdown content
   - Returns array of raw YAML strings
   - Scans the ENTIRE document body regardless of which XML section the block appears in (research recommendation — robust to placement variations)

2. Add `buildGraph(designDir)` function:
   The core graph construction algorithm:

   Step 1: Load events.yaml with js-yaml (via requireYaml()) to get the canonical event registry. events.yaml entries are DEFINITIONS (authoritative names). Parse it and build a Map of event names -> { file, producers: [], consumers: [] }. If events.yaml does not exist, return error: "events.yaml not found at {path} — cannot build graph for Level 4."

   Step 2: For each .md file in agents/ directory:
   - Read the file content
   - Extract frontmatter name field (agent name)
   - Call extractYamlBlocks(body) to get all YAML blocks
   - Parse each block with yaml.load()
   - Look for `dispatches:` arrays (agent produces these events) and `subscribes:` arrays (agent consumes these events)
   - Record: { agentName, file, dispatches: [], subscribes: [] }
   - For each dispatched event: add agent to event's producers list in the graph
   - For each subscribed event: add agent to event's consumers list in the graph

   Step 3: For each .md file in workflows/ directory:
   - Scan content for agent name references (patterns: `agents/{name}.md`, `Task()` calls mentioning agent names, `spawns:` YAML fields)
   - Record which agents are referenced_by each workflow
   - Build spawning edges: if workflow A references agents B and C, record edges A->B and A->C in the spawns adjacency list

   Step 4: Return the complete graph object:
   ```json
   {
     "agents": { "agent-name": { "file": "agents/agent-name.md", "dispatches": [], "subscribes": [], "referenced_by": [], "spawns": [] } },
     "events": { "EventName": { "file": "design/events/events.yaml", "producers": [], "consumers": [] } },
     "edges": { "spawns": [{"from": "A", "to": "B"}], "produces": [{"from": "agent", "to": "event"}], "consumes": [{"from": "agent", "to": "event"}] }
   }
   ```

3. Add `build-graph --design-dir <dir>` CLI command:
   - Calls buildGraph(designDir)
   - Outputs the full graph as JSON to stdout
   - Exit 0 on success, exit 1 if events.yaml missing or parse errors

Update --help and command routing.

<task type="auto">
  <name>Task 2: check-cycles (DFS), find-orphans, and verify level4 commands</name>
  <files>bin/arch-tools.js</files>
  <action>
Add three more commands that query the graph built by buildGraph():

1. `detectCycles(graph)` function — in-house DFS cycle detection (~35 lines):
   - Build adjacency list from spawns edges ONLY (events don't create cycles — only agent-spawns-agent relationships do)
   - Standard DFS with visited Set and recursionStack Set
   - For each agent node: if not visited, run DFS
   - DFS(node, path): add to visited + recursionStack, for each neighbor: if not visited recurse, if in recursionStack CYCLE FOUND — extract cycle from path
   - Remove node from recursionStack on backtrack
   - Return: `{ cycles_found: N, cycles: [{ cycle: ["A", "B", "C", "A"], description: "Circular dependency: A spawns B spawns C spawns A" }] }`
   - Note: This is pre-approved in STATE.md. Do NOT use the toposort npm package unless this implementation fails on edge cases. In-house DFS is preferred for the small agent graph (<20 nodes).

2. `check-cycles --design-dir <dir>` CLI command:
   - Calls buildGraph(designDir) then detectCycles(graph)
   - Outputs cycles JSON to stdout
   - Exit 0 always (cycles are findings, not errors)

3. `findOrphans(graph)` function:
   - Iterate graph.agents: any agent with empty referenced_by array -> orphaned_agent finding
   - Iterate graph.events: any event with empty producers array -> orphaned_event_no_producer finding
   - Iterate graph.events: any event with empty consumers array -> orphaned_event_no_consumer finding
   - Each finding uses the 6-field structured format: { anti_pattern, severity: "blocker", file, entity, detail, remediation }

4. `find-orphans --design-dir <dir>` CLI command:
   - Calls buildGraph(designDir) then findOrphans(graph)
   - Outputs orphans JSON to stdout

5. `verify level4 --design-dir <dir>` command — the comprehensive Level 4 runner:
   Runs ALL Level 4 checks from verification-patterns.md:

   - Check 4a (event_resolves): For every event name referenced in agent spec YAML blocks (dispatches/subscribes), check it exists in events.yaml. Unresolved = orphan reference.
   - Check 4b (agent_resolves): For every agent name referenced in workflow files and orchestration docs, check that agents/{name}.md exists. Unresolved = orphan reference.
   - Check 4c (no_cycles): Run detectCycles() on the spawns graph.
   - Check 4d (no_orphans): Run findOrphans() on the full graph.
   - Check 4e (name_matches_file): For every agent spec, verify that frontmatter `name:` field matches the filename without .md extension.

   Return JSON:
   ```json
   {
     "status": "passed | gaps_found",
     "level": 4,
     "findings": [...],
     "graph_stats": { "agents": N, "events": M, "edges": P }
   }
   ```

Also update the `verify run` command (from 04-02) to call verifyLevel4 when --levels includes 4, instead of the placeholder message.

Update --help and command routing for all new commands.
  </action>
