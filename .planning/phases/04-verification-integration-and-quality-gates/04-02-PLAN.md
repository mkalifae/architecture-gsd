   - Check fs.existsSync(filePath)
   - Return JSON: `{ status: "passed"|"gaps_found", level: 1, file: filePath, findings: [] }`
   - If file missing: findings = [{ check: "file_exists", result: "fail", detail: "File not found: {path}" }]

3. Add `verify level2 <file> [--type agent|schema|failure|topology]` command:
   - Requires file to exist (Level 1 implicit)
   - Check 2a: line count vs minimum by type (agent: 50, schema: 20, failure: 40, topology: 30, default: 50)
   - Check 2b: required sections present
     - For type=agent: check for XML tags: <role>, <upstream_input>, <downstream_consumer>, <execution_flow>, <structured_returns>, <failure_modes>, <constraints> (7 tags). CRITICAL: use XML tags NOT ## headers per STATE.md decision [03-01]
     - For type=schema: check for YAML top-level keys: name:, type:, version:, payload:, error_cases:
     - For type=failure: check for ## headers: "## Failure Mode Catalog", "## Integration Point Failures", "## Residual Risks"
     - For type=topology: check for presence of topology-specific content (Mermaid graph block, channel table, YAML adjacency)
   - Check 2c: no stub phrases — reuse existing scanFileForStubs() function
   - Check 2d: frontmatter required fields
     - For type=agent: name, description, tools, model, color (5 fields)
     - For type=schema: name (at minimum)
   - Return JSON: `{ status: "passed"|"gaps_found", level: 2, file: filePath, findings: [...] }`
   - Each finding: `{ check: "line_count"|"required_sections"|"stub_phrases"|"frontmatter_fields", result: "pass"|"fail", detail: "..." }`

4. Add `scan-antipatterns <file>` and `scan-antipatterns --dir <dir>` commands:
   The 7 anti-patterns from VERF-08:
   - TBD sections: reuse scanFileForStubs (already implemented)
   - Missing failure modes: agent spec missing <failure_modes> section or section content is too short (< 3 named failure modes — count occurrences of "FAILURE-" pattern)
   - Untyped event fields: in YAML files, scan for `type: any` or `type: object` without nested subfields
   - Orphaned agents: agent spec in agents/ not referenced in any workflow file in workflows/ (grep for agent name across workflows/)
   - Orphaned events: event name in events.yaml not referenced by any agent spec dispatches:/subscribes: block (requires reading events.yaml with js-yaml — use requireYaml() lazy loader)
   Note: Undefined references and circular dependencies are Level 4 checks handled in plan 04-03.

   Each finding must have ALL 6 fields (VERF-08 requirement):
   `{ anti_pattern: string, severity: "blocker"|"warning", file: string, entity: string, detail: string, remediation: string }`

   Return JSON: `{ findings: [...], total: N }`

<task type="auto">
  <name>Task 2: verify level3 (cross-reference checks) and verify run (multi-level runner)</name>
  <files>bin/arch-tools.js</files>
  <action>
Add two more commands to bin/arch-tools.js:

1. `verify level3 <file> --design-dir <dir>` command:
   Implements the 4 cross-reference checks from verification-patterns.md Level 3:

   - Check 3a (agent_referenced): For agent specs — grep for the agent name (from frontmatter `name:` field) across all workflow files in workflows/ and orchestration docs. Agent must appear in at least one workflow.
   - Check 3b (event_has_producer_and_consumer): For event schema files — extract event name, then grep across all agent spec files for that event name in dispatches: and subscribes: YAML blocks. Must have at least one producer AND one consumer. Use requireYaml() to parse events.yaml for event names if checking the registry file itself.
   - Check 3c (failure_modes_linked): For failure mode documents — check that the corresponding agent spec references the failure mode document path or the failure mode component name.
   - Check 3d (context_referenced): For CONTEXT.md — check that .arch/STATE.md references CONTEXT.md path.

   Determine which check to apply based on file path:
   - agents/*.md -> check 3a
   - design/events/*.yaml or design/schemas/*.yaml -> check 3b
   - design/failure-modes/*.md -> check 3c
   - .arch/CONTEXT.md -> check 3d

   Return JSON: `{ status: "passed"|"gaps_found", level: 3, file: filePath, findings: [...] }`
   Each finding: `{ check: "agent_referenced"|"event_has_producer"|"event_has_consumer"|"failure_modes_linked"|"context_referenced", result: "pass"|"fail", detail: "...", file: "..." }`

2. `verify run <file|dir> --levels 1,2,3[,4]` command:
   Multi-level verification runner that:
   - Parses --levels argument as comma-separated list of integers (default: 1,2,3)
   - If given a single file: runs each requested level in order, stopping if a level fails (cumulative — must pass Level N before Level N+1)
   - If given a directory: finds all .md and .yaml files, runs each requested level on each file
   - Aggregates results into summary format:
   ```json
   {
     "status": "passed | gaps_found",
     "levels_run": [1, 2, 3],
     "summary": {
       "level_1": { "passed": 12, "failed": 0 },
       "level_2": { "passed": 10, "failed": 2 },
       "level_3": { "passed": 8, "failed": 2 }
     },
     "findings": [...]
   }
   ```
   - Auto-detects document type from file path for Level 2 (agents/ -> agent, design/events/ -> schema, design/failure-modes/ -> failure, design/topology/ -> topology)
   - Level 4 is accepted in --levels but returns a placeholder message "Level 4 requires build-graph — use verify level4 command directly" (Level 4 implementation is in plan 04-03)

Update --help text and command routing for both new commands.
  </action>
