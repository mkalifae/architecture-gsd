  <name>Task 1: verify-phase workflow — full verification pipeline orchestrator</name>
  <files>workflows/verify-phase.md</files>
  <action>
Replace the Phase 1 stub in workflows/verify-phase.md with a complete verification pipeline workflow. Preserve the existing frontmatter (description, allowed-tools, argument-hint) but update the description to be more specific.

The workflow must be STANDALONE — invoked manually by the human via `/arch-gsd:verify-phase N`. It is NOT auto-invoked from execute-phase (research recommendation: verification is a phase sign-off gate, not a step in execution, matching the GSD pattern).

Implement a 9-step orchestration pipeline:

Step 1: Parse phase argument. Read .arch/STATE.md for current position. Validate that the requested phase number is valid and that the phase directory exists.

Step 2: Prerequisite check — verify that Phase N design execution is complete:
- Check .arch/STATE.md for phase completion status
- Check that design/ directory exists and contains at least the minimum expected documents for the phase
- If prerequisites not met: display "Phase N design documents not found. Run /arch-gsd:execute-phase N first." and stop.
(This prevents the Pitfall 5 from research: verification running on empty directory producing misleading all-Level-1-failure results.)

Step 3: Spawn arch-verifier (Task() with fresh context):
- Pass paths: agents/arch-verifier.md, references/verification-patterns.md, .arch/STATE.md, design/ directory
- arch-verifier runs all 4 levels + anti-pattern scan
- arch-verifier writes design/VERIFICATION.md
- Wait for completion, capture return status

Step 4: Check arch-verifier result:
- If status: "failed" — display failure details, stop workflow
- If status: "human_needed" — display findings requiring human judgment, stop workflow with message to resolve and re-run
- If status: "passed" or "gaps_found" — continue to arch-integrator

Step 5: Spawn arch-integrator (Task() with fresh context):
- Pass paths: agents/arch-integrator.md, design/VERIFICATION.md, design/digests/, .arch/STATE.md
- arch-integrator reads VERIFICATION.md frontmatter, reads digests, runs cross-phase consistency checks
- arch-integrator writes design/INTEGRATION-REPORT.md
- Wait for completion, capture return status

Step 6: Check arch-integrator result:
- If status: "failed" or "human_needed" — display details, stop workflow
- If status: "passed" or "gaps_found" — continue

Step 7: Generate MANIFEST.md:
- Scan design/ directory for ALL output documents
- Build document index with type, verification level, and status (from VERIFICATION.md findings)
- Determine reading order: CONTEXT.md first, then events.yaml, then topology, then agent specs (alphabetical), then failure modes, then context flows
- Write design/MANIFEST.md using the template format
(Research recommendation: MANIFEST.md is produced by verify-phase workflow directly, not by arch-integrator — verify-phase has the complete view of all documents verified.)

Step 8: Write phase-boundary DIGEST.md as the FINAL step:
- Call `node bin/arch-tools.js write-digest --phase N --design-dir design/`
- This must be the LAST step (Pitfall 3 from research: digest written mid-analysis is incomplete because not all documents scanned yet)
- DIGEST.md is a READER TOOL for the NEXT session — it orients quickly at the start of the next phase

Step 9: Report results:
- Display verification summary: VERIFICATION.md status, INTEGRATION-REPORT.md status, findings count
- Display paths to all output artifacts: VERIFICATION.md, INTEGRATION-REPORT.md, MANIFEST.md, DIGEST.md
- Update .arch/STATE.md with verification completion status
- If gaps_found in either report: display recommended actions (re-run execute-phase for fixable gaps, or resolve manually for human_needed findings)

Context discipline (matching execute-phase pattern from 03-06): keep orchestrator context at ~15% by passing paths to subagents, not reading full documents. Subagents (arch-verifier, arch-integrator) get fresh 200K windows.
