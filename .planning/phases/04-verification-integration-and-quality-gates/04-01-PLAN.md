The execution_flow must implement a 7-step verification pipeline:

Step 1: Read .arch/STATE.md to orient — extract current phase identifier and any locked decisions. Read .arch/CONTEXT.md if present for locked-decisions that affect verification expectations.

Step 2: Locate design documents for the current phase. Build the file list by scanning design/ directory (agent specs in design/agents/, event schemas in design/events/, failure modes in design/failure-modes/, topology in design/topology/, context flows in design/context-flows/). If no design documents found, return status: "failed" with detail: "No design documents found for phase N."

Step 3: Run Level 1 checks on all documents — call `node bin/arch-tools.js verify level1 <file>` for each expected file. Collect results. Documents that fail Level 1 are excluded from subsequent levels.

Step 4: Run Level 2 checks on all Level-1-passing documents — call `node bin/arch-tools.js verify level2 <file> --type <doctype>` for each. doctype is agent|schema|failure|topology based on the file's directory location. Collect results. CRITICAL: For agent specs, Level 2 required section check must use XML tags (<role>, <upstream_input>, <downstream_consumer>, <execution_flow>, <structured_returns>, <failure_modes>, <constraints>), NOT ## markdown headers (STATE.md decision [03-01]).

Step 5: Run Level 3 checks on all Level-2-passing documents — call `node bin/arch-tools.js verify level3 <file> --design-dir design/`. Cross-reference checks: agent specs referenced in workflows, events have producers AND consumers, failure modes linked from agent specs.

Step 6: Run Level 4 checks — call `node bin/arch-tools.js verify level4 --design-dir design/`. Full YAML graph consistency: event names resolve against events.yaml canonical registry, agent names resolve against agents/ directory, no circular agent dependencies, no orphaned events or agents, agent name matches filename.

Step 7: Run anti-pattern scan — call `node bin/arch-tools.js scan-antipatterns --dir design/`. Parse JSON findings. Incorporate each finding into the VERIFICATION.md findings array with the structured finding format (anti_pattern, severity, file, entity, detail, remediation).

Step 8: Write VERIFICATION.md — assemble all findings from Steps 3-7 into the canonical VERIFICATION.md format with frontmatter (phase, status, levels_run, timestamp), summary table, individual findings, anti-pattern scan results section, and recommended action. Status rules: passed = zero findings with result=fail, gaps_found = one or more fail findings that are programmatically fixable, human_needed = findings requiring human judgment (semantic inconsistency, conflicting locked-decisions).

Step 9: Return structured result to verify-phase workflow.

The structured_returns section must define 4 return states with literal JSON examples:
- status: "passed" — all levels run, zero findings with result=fail, VERIFICATION.md written with status: passed
- status: "gaps_found" — one or more Level 1-4 findings or anti-pattern findings, VERIFICATION.md written with status: gaps_found, includes findings array
- status: "human_needed" — finding requires human judgment, VERIFICATION.md written with status: human_needed
- status: "failed" — verification could not run (no design documents, arch-tools.js command errors)

The failure_modes section must define at least 3 failure modes with trigger/manifestation/severity/recovery/detection fields:
- FAILURE-01: No design documents found (Level 1 all-fail)
- FAILURE-02: arch-tools.js command returns non-zero exit (tooling failure)
- FAILURE-03: False positive from detect-stubs on instructional content within design documents
- FAILURE-04: Level 4 cannot run because events.yaml does not exist (schema-designer did not run or failed)

Avoid detect-stubs false positives: reference banned phrases via meta-description ("vague escalation language", "stub phrases") rather than quoting them inline.
