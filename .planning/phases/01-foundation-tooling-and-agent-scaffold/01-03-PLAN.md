Create `references/naming-conventions.md` — the canonical naming rules for all Architecture GSD artifacts.

**1. Three naming categories with exact regex patterns:**

| Category | Convention | Regex Pattern | Examples (Valid) | Examples (Invalid) |
|----------|-----------|---------------|------------------|--------------------|
| Events | PascalCase | `^[A-Z][a-zA-Z0-9]*$` | TaskAssigned, PhaseComplete, ContextValidated | task_assigned, TASK_ASSIGNED, task-assigned |
| Commands | SCREAMING_SNAKE_CASE | `^[A-Z][A-Z0-9_]*$` | ASSIGN_TASK, VALIDATE_CONTEXT, START_PHASE | AssignTask, assign_task, assign-task |
| Agent Names | kebab-case | `^[a-z][a-z0-9-]*$` | arch-executor, discuss-system, failure-analyst | ArchExecutor, arch_executor, ARCH_EXECUTOR |

**2. Scope rules — where each convention applies:**
- Agent names: `name:` field in agent spec frontmatter, agent references in workflow docs, keys in config.json model_profiles
- Event names: `name:` field in event-schema.yaml files where `type: event`, event references in agent specs (dispatches/subscribes sections)
- Command names: `name:` field in event-schema.yaml files where `type: command`, command references in orchestration docs

**3. Compound naming rules:**
- File names for agent specs: `{agent-name}.md` (kebab-case, matching the name frontmatter field)
- File names for event schemas: `{EventName}.yaml` or collected in `events.yaml` (PascalCase per event within the file)
- Phase directory names: `{NN}-{slug}` where slug is kebab-case derived from phase name
- Design document file names: kebab-case with `.md` or `.yaml` extension

**4. Programmatic enforcement:**
- State that `arch-tools.js validate-names` enforces these rules
- Include the exact regex patterns that validate-names uses
- Note: validation scans frontmatter `name:` fields and YAML `- event:` references

**5. Rationale section:**
- PascalCase events: Matches industry standard (CloudEvents, domain events in DDD). Visually distinct from agent names and commands.
- kebab-case agents: Matches Claude Code convention for agent file names. URL-safe. Naturally maps to file names.
- SCREAMING_SNAKE commands: Visually distinct from events. Signals imperative intent (DO_SOMETHING vs SomethingHappened).

Do NOT make this a vague guideline document. Include the exact regex patterns, exact scope rules, and exact examples. This is a machine-checkable contract, not a style guide suggestion.

  <name>Task 2: Create CONTEXT.md schema specification document</name>
  <files>references/context-schema.md</files>
  <action>
Create `references/context-schema.md` — the complete schema specification for CONTEXT.md files that discuss-system produces in Phase 2.

This document must contain:

**1. Required frontmatter fields (all 6):**

| Field | Type | Required | Non-Empty | Description |
|-------|------|----------|-----------|-------------|
| `domain` | string | yes | yes (non-blank) | What kind of system: "event-driven multi-agent", "microservices", etc. |
| `actors` | array of strings | yes | yes (min 1 item) | Human or system actors who interact with the system |
| `non-goals` | array of strings | yes | yes (min 1 item) | Explicit list of what the system does NOT do — MUST NOT be empty |
| `constraints` | array of strings | yes | yes (min 1 item) | Technical, operational, or organizational constraints |
| `scale` | object | yes | n/a (structured) | System scale parameters (see nested structure) |
| `locked-decisions` | array of objects | yes | n/a | Architectural decisions already made |

**2. Nested field structures:**

`scale` object fields:
```yaml
scale:
  agents: integer or string    # Count or range (e.g., 5 or "10-20")
  throughput: string            # Transactions/second or similar (e.g., "1000 req/s")
  latency: string              # Target latency (e.g., "< 200ms p99")
```

`locked-decisions` array item structure:
```yaml
locked-decisions:
  - decision: string           # What was decided
    rationale: string          # Why this decision was made
```

**3. Required body sections:**

| Section | Required | Content Expectation |
|---------|----------|---------------------|
| `## System Intent` | yes | Natural language description, min 3 sentences |
| `## Actors` | yes | Expanded description of each actor from frontmatter |
| `## Non-Goals` | yes | Each non-goal explained with reasoning |
| `## Constraints` | yes | Hard limits the design must honor |
| `## Scale` | yes | Volume, throughput, latency targets with reasoning |
| `## Locked Decisions` | yes | Each decision with full rationale |

**4. Validation rules (what `validate-context` checks):**
- All 6 frontmatter fields must be present (not undefined/null)
- `domain` must be a non-empty string (not "" or whitespace-only)
- `actors` must be an array with >= 1 item
- `non-goals` must be an array with >= 1 item
- `constraints` must be an array with >= 1 item
- `scale` must be an object (not a string or array)
- `locked-decisions` must be an array (can be empty if no decisions are locked yet)

**5. Complete example CONTEXT.md** — A realistic example showing a well-formed CONTEXT.md for a hypothetical "task orchestration system". Include all frontmatter fields populated correctly and all body sections with substantive content. This example serves as the reference for discuss-system's output format.

**6. Anti-patterns:**
- `non-goals: []` — empty non-goals is invalid (every system has things it explicitly does NOT do)
- `domain: "software"` — too vague, must describe the system type
- `constraints: ["be fast"]` — too vague, must be specific and measurable
- `scale: "large"` — must be structured object, not a flat string (per research recommendation)

**7. Relationship to validate-context:**
Explicitly state that `arch-tools.js validate-context` enforces this schema. The validate-context command checks frontmatter field presence and non-emptiness. Body section validation is deferred to Level 2 verification (arch-verifier in Phase 4).

Do NOT define a partial schema — this must be the COMPLETE schema that Phase 2 will produce against. Otherwise Phase 2 will need to revise Phase 1 artifacts.
