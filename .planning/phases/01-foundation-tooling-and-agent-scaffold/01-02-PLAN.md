Create `bin/arch-tools.js` as a Node.js CLI utility with `#!/usr/bin/env node` shebang. Use ONLY Node.js built-ins (`fs`, `path`, `child_process`). Zero npm dependencies.

**Frontmatter CRUD commands (FOUN-02):**

Copy the frontmatter extraction/reconstruction approach from GSD's `gsd-tools.js` (lines 248-385). The regex-based YAML parser handles the simple key:value, array, and nested object patterns used in agent specs and plan files.

- `frontmatter get <file> [--field key]` — Read file, extract YAML frontmatter between `---` delimiters, output as JSON. If `--field` specified, output just that field's value.
- `frontmatter set <file> --field key --value jsonVal` — Parse existing frontmatter, update specified field, reconstruct frontmatter, write back to file preserving body content.
- `frontmatter merge <file> --data '{json}'` — Parse existing frontmatter, merge JSON object into it (shallow merge), reconstruct, write back.

Output format: `{ frontmatter: {...}, file: "path" }` for get, `{ updated: true, field: "key", file: "path" }` for set, `{ merged: true, fields: [...], file: "path" }` for merge.

**Stub detection command (FOUN-03):**

- `detect-stubs <file>` — Scan file body (AFTER stripping frontmatter) for stub phrases.
- `detect-stubs --dir <directory>` — Scan all `.md` files in directory recursively.

Stub patterns to detect (case-insensitive except where noted):
```javascript
const STUB_PATTERNS = [
  { pattern: /\bTBD\b/i, label: 'TBD' },
  { pattern: /to be determined/i, label: 'to be determined' },
  { pattern: /handles gracefully/i, label: 'handles gracefully' },
  { pattern: /as needed/i, label: 'as needed' },
  { pattern: /will be (added|defined|specified) later/i, label: 'deferred specification' },
  { pattern: /placeholder/i, label: 'placeholder' },
  { pattern: /coming soon/i, label: 'coming soon' },
  { pattern: /TODO\b/, label: 'TODO' },
  { pattern: /FIXME\b/, label: 'FIXME' },
  { pattern: /details to follow/i, label: 'details to follow' },
  { pattern: /specify later/i, label: 'specify later' },
];
```

IMPORTANT: Strip frontmatter before scanning — only scan the document body. Frontmatter may legitimately contain placeholder values during drafting.

Output format for single file:
```json
{ "file": "path", "stubs_found": 2, "stubs": [{ "line": 14, "text": "TBD", "pattern": "TBD" }], "clean": false }
```

Output format for directory:
```json
{ "directory": "path", "files_scanned": 5, "files_with_stubs": 2, "results": [...] }
```

Also include a `--help` handler that prints all available commands and their usage. The help text should list every command group (frontmatter, detect-stubs, state, validate-context, validate-names) with argument syntax.

**Phase state management commands (FOUN-04):**

- `state init --phase N --name <name>` — Create phase directory at `.planning/phases/{NN}-{slug}/`, update STATE.md to record current phase. Output: `{ phase_dir: "...", padded_phase: "01", created: true, state_updated: true }`

- `state transition --from N --to M` — Record phase transition. Update STATE.md current phase from N to M. Output: `{ from: N, to: M, transitioned: true }`

- `state status [--phase N]` — Query current phase position from STATE.md. If `--phase N`, return status for specific phase. Output: `{ current_phase: N, phase_name: "...", status: "in_progress|complete|not_started" }`

State management should read/write `.planning/STATE.md` using regex-based field updates (find line with `Phase:`, replace value). Create STATE.md if it doesn't exist.

**CONTEXT.md validation command (FOUN-06):**

- `validate-context <file>` — Validate that a CONTEXT.md file has all 6 required frontmatter fields, and that list fields are non-empty.

Required fields: `domain`, `actors`, `non-goals`, `constraints`, `scale`, `locked-decisions`
Non-empty list fields: `actors`, `non-goals`, `constraints` (must be arrays with at least one item)
Non-empty string field: `domain` (must be non-blank string)

Output format:
```json
{
  "valid": false,
  "missing_fields": ["non-goals", "scale"],
  "empty_required_fields": ["actors"],
  "present_fields": ["domain", "constraints", "locked-decisions"],
  "file": "path"
}
```

**Naming convention validation command (FOUN-05 programmatic enforcement):**

- `validate-names <file>` — Scan a file for naming convention violations.
- `validate-names --dir <directory>` — Scan all files in directory.

Naming rules:
- Agent names (frontmatter `name:` field): must match `/^[a-z][a-z0-9-]*$/` (kebab-case)
- Event names (in YAML `name:` fields or `- event:` references): must match `/^[A-Z][a-zA-Z0-9]*$/` (PascalCase)
- Command names (in YAML `name:` fields where `type: command`): must match `/^[A-Z][A-Z0-9_]*$/` (SCREAMING_SNAKE)

Output format:
```json
{
  "file": "path",
  "violations": [
    { "type": "agent", "name": "ArchChecker", "rule": "kebab-case", "expected_pattern": "^[a-z][a-z0-9-]*$", "example": "arch-checker" }
  ],
  "valid": true
}
```

Also add a `loadConfig` function that reads `config.json` from the project root and resolves model profiles. This mirrors GSD's `loadConfig` pattern. The function should return the parsed config with defaults applied for any missing fields.
