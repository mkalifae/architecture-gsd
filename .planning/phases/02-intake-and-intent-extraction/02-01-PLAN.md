**<execution_flow> section — the core questioning strategy:**

Implement a 10-step execution flow modeled on GSD's discuss-phase.md pattern, adapted for architecture intake:

Step 1: Read references/context-schema.md to load the schema contract (all 6 fields, types, validation rules).

Step 2: Read the system description argument passed by new-system.md workflow.

Step 3: Pre-analyze the system description. Draft working hypotheses for each CONTEXT.md field:
  - domain: what architectural category?
  - actors: who interacts with this system?
  - non-goals: what is mentioned as out-of-scope? (often nothing — flag as gap)
  - constraints: what platform/tech limits are stated?
  - scale: what volume/throughput/latency targets? (often nothing — flag as gap)
  - locked-decisions: any explicit architectural choices already made?

Step 4: Identify gray areas — fields that are absent, vague, or ambiguous in the description. Every field with a gap is a gray area.

Step 5: Present gray areas as a grouped multi-select menu. CRITICAL: non-goals must appear in the menu and be marked as "(REQUIRED — cannot be empty)" even if the description mentioned something about scope. If ALL 6 fields can be populated with concrete values (including at least one non-goal), present a CONFIRMATION menu instead ("I've extracted the following — confirm or modify").

Step 6: For each selected gray area, ask 3-4 concrete questions with multiple-choice options where possible. Questions must produce concrete values (not vague adjectives) for each CONTEXT.md field.
  - Scale questions MUST ask for agents (number or range), throughput (rate + unit), and latency (bound + unit) as three separate sub-questions.
  - Non-goals questions MUST present domain-appropriate exclusion candidates the human can confirm or reject.
  - Locked-decisions questions MUST ask "Have you already made any architectural choices that the design must honor?"

Step 7: Pre-flight check before writing: verify non-goals has at least one item. If no non-goals collected: surface the requirement explicitly ("Before we proceed, I need at least one explicit non-goal. Every system has things it deliberately doesn't do. What does this system NOT handle?"). Do not proceed to writing until at least one non-goal exists.

Step 8: Write .arch/CONTEXT.md with all collected values per the schema in context-schema.md. Use YAML frontmatter for the 6 machine-checkable fields. Write all 6 body sections (## System Intent, ## Actors, ## Non-Goals, ## Constraints, ## Scale, ## Locked Decisions) with substantive prose, not just field echoes.

Step 9: Validate by running: `node bin/arch-tools.js validate-context .arch/CONTEXT.md`. If validation fails: correct the issue inline (add missing fields, fix type errors). Re-validate until clean. Do not return until validate-context returns `valid: true`.

Step 10: Return structured status to the spawning workflow.

**<failure_modes> section:**

Define exactly 4 failure modes following templates/failure-modes.md format:

FAILURE-01: Human abandons conversation mid-flow
  - Trigger: No response after presenting gray area menu
  - Manifestation: Cannot produce complete CONTEXT.md
  - Recovery: Write partial CONTEXT.md with collected fields, mark missing fields with TBD comments. Return status "partial" with fields_missing list.

FAILURE-02: System description too vague (fewer than 20 words, no domain signals)
  - Trigger: Cannot draft working hypotheses for any field
  - Manifestation: Would need to ask about all 6 fields from scratch
  - Recovery: Skip pre-analysis. Open with freeform prompt: "Tell me about the system — what does it do, who uses it, and what technology environment does it run in?" Then continue to gray area menu.

FAILURE-03: validate-context fails after 2 correction attempts
  - Trigger: Writing CONTEXT.md, correcting, re-validating — still fails
  - Manifestation: Type errors or structural errors that persist
  - Recovery: Return status "human_needed" with error details and partial CONTEXT.md path. Do not silently write invalid CONTEXT.md and signal success.

FAILURE-04: Human provides contradictory locked-decisions
  - Trigger: Two decisions that conflict (e.g., "use event sourcing" and "stateless — no persistent state")
  - Manifestation: Locked decisions that cannot both be honored
  - Recovery: Surface the contradiction explicitly and ask which takes precedence. Do not write contradictory locked-decisions.
