**<purpose> section:**

One command takes the human from natural language description to a fully validated CONTEXT.md ready for the design pipeline.

**<process> section with 6 steps:**

**Step 1: Setup and Validation**
- Extract system description from $ARGUMENTS.
- If $ARGUMENTS is empty: support no-argument mode. Display a prompt asking the human to describe their system ("Describe the agentic system you want to design — what it does, who uses it, and at what scale"). Use the response as the system description going forward.
- Check if .arch/CONTEXT.md already exists. If it does, present 3 options via user interaction:
  - "Update it" — re-run discuss-system with existing CONTEXT.md as starting point
  - "View it" — display current CONTEXT.md, then offer update/skip
  - "Skip" — use existing CONTEXT.md, jump to Step 5 (STATE.md initialization)

**Step 2: Scaffold Project Directory**
- Create .arch/ directory: `mkdir -p .arch`
- Do NOT use `arch-tools.js state init` here — that command creates directories under `.planning/phases/`, not `.arch/`. The `.arch/` directory is created by `mkdir -p` alone. STATE.md is written directly by the Write tool in Step 5.

**Step 3: Spawn discuss-system**
- Display status message: "Spawning discuss-system... Analyzing system description and identifying clarifying questions."
- Use Task() to spawn discuss-system as subagent:
  - prompt: Read agents/discuss-system.md for instructions. Pass system description. Reference @references/context-schema.md. Instruct to produce .arch/CONTEXT.md and validate via arch-tools.js validate-context.
  - model: opus (per config.json balanced profile)

**Step 4: Handle discuss-system Return**
- Parse the structured return from discuss-system.
- IF status is "human_needed": display the reason and partial_context path. Instruct user to re-run /arch-gsd:new-system after addressing the issue. Stop.
- IF status is "complete": continue to Step 5.
- Safety net: run `node bin/arch-tools.js validate-context .arch/CONTEXT.md` regardless of reported status. If it fails despite "complete" status, treat as "human_needed".

**Step 5: Initialize STATE.md**
- Read locked-decisions from CONTEXT.md: `node bin/arch-tools.js frontmatter get .arch/CONTEXT.md --field locked-decisions`
- Read domain from CONTEXT.md for the summary line: `node bin/arch-tools.js frontmatter get .arch/CONTEXT.md --field domain`
- Write .arch/STATE.md using the Write tool. The STATE.md template must be 50-70 lines and contain EXACTLY these three required sections:
  - `# Project State` header
  - `## Project Reference`: "See: .arch/CONTEXT.md (produced {date})" + core value summary (from discuss-system's system_intent_summary) + current focus "Phase 1 — [first design phase]"
  - `## Current Position`: Phase 0 (Intake complete), Plan N/A, Status "CONTEXT.md produced and validated", Last activity date
  - `## Accumulated Context` > `### Decisions`: extracted locked-decisions (each as a bullet). If empty: "No decisions locked at intake — design pipeline has full latitude"
  - `### Pending Todos`: None
  - `### Blockers/Concerns`: None
  - `## Session Continuity`: Last session date, Stopped at "Intake complete", Resume with "/arch-gsd:execute-phase 1"
- CRITICAL: STATE.md must NOT exceed 70 lines. It is an index, not a copy of CONTEXT.md. Full content lives in CONTEXT.md.

**Step 6: Confirm and Display Next Steps**
- Commit both files using direct git commands: `git add .arch/CONTEXT.md .arch/STATE.md && git commit -m "feat: initialize architecture project"`
- Display confirmation:
  ```
  Architecture GSD -- Project Initialized

  System: [domain from CONTEXT.md]
  Context: .arch/CONTEXT.md (validated)
  State:   .arch/STATE.md (initialized)

  Locked decisions: [count] decisions locked at intake
  Non-goals: [count] explicit non-goals

  Next: /arch-gsd:execute-phase 1 (after /clear for fresh context)
  ```

**Responsibility boundary enforcement:**
- new-system.md must NOT contain any AskUserQuestion calls about system content (domain, actors, scale, etc.) — all content questioning belongs in discuss-system.
- new-system.md MAY use AskUserQuestion only for workflow-level decisions (update/view/skip existing CONTEXT.md, no-argument mode prompt).
- new-system.md must NOT write CONTEXT.md — that is discuss-system's job.
- new-system.md MUST write STATE.md — that is the orchestrator's responsibility after discuss-system returns.
